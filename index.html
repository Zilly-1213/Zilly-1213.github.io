2022.5.19

今天是个带晴天，hzl闲的蛋疼决定重新搭一个博客。虽然服务器在国外就很蛋疼，不过用来放自己日常生活、学习的记录足够了。

下面简单记录一下自己搭建该博客的过程和坑吧。

1. 环境准备

一个博客需要什么？一个有趣的灵魂和存放灵魂的容器。现在一个有趣的灵魂已经有了，那么就需要寻找足够配得上这个有趣灵魂的容器。

那么怎么评价一个容器够不够优秀呢，首先要看它的性能（耐不耐折腾），其次看它的外表（好看才是王道！），最后看它的价格。（白嫖万岁！）

现在hzl找到了一个性价比无穷大的容器（因为它完全不需要花钱），它基于github仓库+hexo博客框架。

首先看github仓库的配置：

  1. 新建一个仓库命名为zily-1213.github.io，必须要这样命名
  2. 在命令行使用指令ssh-keygen -t rsa -C "注释文字"生成SSH秘钥。然后在github用户的setting里添加SSH秘钥，这样子就实现了可以从命令行对github的加密登录。（SSH可以理解为一种网络协议，允许计算机以设置好的用户身份加密链接到远程服务器）

然后是Hexo的安装与配置

  1. 在终端运行指令npm install -g hexo-cli进行hexo的全局安装。
  2. 创建一个blog文件夹，用于存放hexo框架、博客文章等文件。
  3. 终端运行npm指令生成相关依赖环境。
  4. 在文件夹中执行hexo init命令，生成hexo相关文件，最重要的是source文件夹，其中存放的是会被post到github仓库中的博客文件。
  5. hexo相关指令操作见参考的博客：https://segmentfault.com/a/1190000038619366

然后我们的博客就已经具备雏形啦，bravo！

2.图床的搭建

很久很久以前，在遥远东方大陆的程序猿们经常会去一片名为Github的香蕉园采香蕉吃，这片香蕉园的香蕉非常多，而且没有人管制，（也有人说有一个名叫开源的管理员在管理），想吃多少吃多少，许多小程序猿就在采香蕉，吃香蕉，采香蕉的过程中慢慢成为了大程序猿。但是有一天，不知道什么时候香蕉园开始变换了位置，一会再北边，一会在南边，变幻莫测，这让程序猿们没法按时正常吃香蕉了，每次找香蕉园就要花很多时间（有人说是一股名叫SSL的神秘力量在搞鬼）

没办法，谁让hzl的博客是基于github搭建的呢，于是hzl为了在访问博客时节省图片等静态资源的加载时间，决定采用图床存储图片的方案。

Github + Picgo搭建的图床

hzl本着能白嫖就白嫖的想法，想到了使用github作为图片存储仓库，Picgo作为图片传输接口与URl转换的工具，jsDelivr作为CDN缓存服务器，从而搭建一个免费的图床，同时又免费的CDN加速，岂不美哉？

  1. 首先创建一个github仓库，用于存放上传的图片。
  2. 在github的setting中生成Access token即访问令牌。
  3. 配置PicGo，填入仓库名、分支名，填入token，设定自定义域名为jsDelivr的格式。

接下来就可以愉快地使用图床啦！只需要把图片往PicGo一拖，图片就能够上传到github仓库中，同时会在免费的CDN服务器中缓存，perfect！

然而真正测试的时候hzl傻了，爷的图图呢？！

hzl小小的脑袋装满了大大的疑惑，突然想到了什么。之前说github的位置会变换，所以访问速度会变慢，而我的图床又是以github仓库为存储容器的，所以我在访问图片时还是需要以龟速先访问一次github这个神出鬼没的香蕉园，其次这个白嫖的CDN服务器似乎也没有正常的给我缓存图片资源，每次加载图片要么加载不出来，要么是龟速。。。好吧看来我还是太天真了。

经过友人周大仙的推荐，hzl决定以国内能够访问的图床sm.ms+Picgo来搭建自己的图床环境。而sm.ms本身提供5G的免费试用空间，四舍五入也是半永久白嫖了，hooray！

设置sm.ms图床需要几步？

  2. 在Picgo导入sm.ms的token

这么大的工程可比把大象塞进冰箱还少一步呢！
